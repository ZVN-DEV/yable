## Summary

The redesign is structurally solid. The 5-section layout is correct, the architectural split (server page.tsx + client HomeClient.tsx) is clean, and all deleted sections (packageBand, themeNotes, fieldNotes, showcase sidebar, HeroPreview) are gone. SEO foundations are in place. The punch list below captures content-copy mismatches against the content writer's final deliverable, a layout.tsx background-color discrepancy, extra shadows beyond spec allowance, a missing Speed section eyebrow, and CTA button label drift.

## Punch List

### Critical (blocks sign-off)

- [ ] PM-1: **CTA body copy does not match content writer.** Content writer specifies: "Zero to interactive table in 11 steps. Each step has copy-paste code verified in a live Next.js app." Implementation at `HomeClient.tsx:767` reads: "Zero to interactive table in 11 steps. Copy-paste code, tested in Next.js." The second sentence has been paraphrased and loses the "each step" + "verified" language. **Fix: use verbatim content-writer copy.**

- [ ] PM-2: **CTA button labels do not match content writer.** Content writer specifies: "Quickstart -->", "Features", "Async Commits". Implementation at `HomeClient.tsx:771-779` reads: "Start the quickstart -->", "Browse features", "Async commits guide". All three labels are paraphrased. **Fix: use "Quickstart -->", "Features", "Async Commits" exactly.**

- [ ] PM-3: **Speed section description does not match content writer.** Content writer specifies: "Pretext measures every variable-height row before first paint. Pick a dataset -- numbers are live." Implementation at `HomeClient.tsx:842` reads: "Canvas-measured, prefix-sum indexed. Live numbers -- pick a dataset." This is developer jargon, not the approved copy. **Fix: use verbatim content-writer copy.**

- [ ] PM-4: **Speed section missing eyebrow.** Content writer specifies eyebrow: "Speed". The `SpeedShowcase` component at `HomeClient.tsx:838-844` has an `<h2>` but no `<span className={s.sectionEyebrow}>Speed</span>` before or above the title. Every other section has its eyebrow. **Fix: add `<span className={s.sectionEyebrow}>Speed</span>` in the speedHead div, before the h2.**

- [ ] PM-5: **Crown Jewels eyebrow capitalization mismatch.** Content writer specifies: "What Ships Free" (title case). Implementation at `HomeClient.tsx:708` reads: "What ships free" (sentence case). The CSS applies `text-transform: uppercase` so visually it renders as "WHAT SHIPS FREE" in all cases, but the source should match the canonical copy for consistency and in case CSS changes. **Demoted to Important if team confirms uppercase-only rendering is permanent.**

### Important (should fix)

- [ ] PM-6: **layout.tsx body backgroundColor does not match --bg-0.** Spec section 7.2 defines `--bg-0: #0a0706`. The CSS `.page` class correctly uses `--bg-0: #0a0706` at `page.module.css:7`. But `layout.tsx:58` sets inline `backgroundColor: '#0d0907'` on the `<body>`. This creates a flash of wrong-colored background before the CSS loads, and the hex value `#0d0907` does not appear anywhere in the spec. **Fix: change to `#0a0706` to match `--bg-0`.**

- [ ] PM-7: **Hero GitHub CTA label says "View on GitHub" instead of "GitHub".** Content writer specifies CTA secondary label: "GitHub". Implementation at `HomeClient.tsx:653` reads: "View on GitHub". While more descriptive, it deviates from approved copy. **Fix: change to "GitHub" or get explicit sign-off on the expanded version.**

- [ ] PM-8: **Extra shadows beyond spec allowance.** Spec section 7.4 says: "Shadows: none on the page surface. One subtle shadow allowed on the hero code block." The hero code block shadow at `page.module.css:317` is correct. But additional shadows exist at: (a) `page.module.css:128` -- kicker dot `box-shadow: 0 0 10px` (decorative glow), (b) `page.module.css:908` -- pending dot active `box-shadow: 0 0 0 4px` (functional indicator), (c) `page.module.css:1072-1077` -- theme card hover/selected `box-shadow: 0 16px 36px` and `box-shadow: 0 0 0 1px`. The theme card hover shadow at line 1072 is the most visible violation -- it adds a large drop shadow on hover. **Fix: remove the theme card hover shadow; the kicker glow and pending-dot indicator are borderline and can be waived if desired.**

- [ ] PM-9: **Feature card copy uses plain text where content writer uses backtick-formatted code.** Content writer specifies: "`SUM`, `IF`, `ROUND`" and "`onCommit`" with inline code formatting. Implementation at `HomeClient.tsx:184` renders these as plain text strings without any `<code>` wrapping. The feature card code snippet below partially compensates, but the inline references in the copy paragraph lose their monospace treatment. **Fix: wrap API names in `<code className={s.inlineCode}>` tags within the feature copy.**

- [ ] PM-10: **Hero primary CTA label says "Read the docs -->" instead of "Read the docs -->".** Content writer specifies: "Read the docs -->". Implementation at `HomeClient.tsx:644-645` reads: "Read the docs" + a separate `<span aria-hidden>-->`. This is functionally equivalent and accessible. No fix needed, but noting for completeness. **Status: acceptable.**

- [ ] PM-11: **page.tsx meta description exceeds 160 chars.** Spec S6 says "140-160 char description". The `page.tsx:8` description is: "The TypeScript-first React data grid with pivot tables, formula engine, fill handle, clipboard, undo/redo, and async cell commits. All MIT-licensed, zero runtime deps. The AG Grid alternative that ships free." That is ~207 characters. The content writer provided a 150-character version: "Yable: open-source React data table with pivot tables, formulas, fill handle, clipboard, undo/redo, and async commits. TypeScript-first. MIT-licensed." **Fix: use the content writer's 150-char meta description.**

- [ ] PM-12: **layout.tsx metadata description also too long / different from page.** `layout.tsx:32` has a separate 149-char description which is fine as the global fallback. But the `page.tsx` description (the one that actually renders for `/`) should be the content writer's version per PM-11. No additional fix needed on layout.tsx.

### Minor (nice to have)

- [ ] PM-13: **Hero grid ratio is 1.05fr / 0.95fr (52.5/47.5) instead of spec's 55/45.** Spec section 6.1 says "2-col 55/45". `page.module.css:77` uses `minmax(0, 1.05fr) minmax(0, 0.95fr)` which works out to ~52.5/47.5. The visual difference is small. **Suggest: change to `minmax(0, 1.1fr) minmax(0, 0.9fr)` for a closer 55/45 split, or keep as-is with explicit sign-off.**

- [ ] PM-14: **CSS custom properties scoped to `.page` instead of `:root`.** Spec section 7.2 shows the variables under `:root`. Implementation puts them in `.page` (page.module.css:7-19). This works because `.page` wraps everything on the landing, but means the variables are unavailable in `:root` scope and could cause issues if any child component references them outside the `.page` wrapper. **Low risk; note for awareness.**

- [ ] PM-15: **"What Ships Free" eyebrow casing vs. CSS uppercase.** As noted in PM-5, the source text uses sentence case while CSS applies `text-transform: uppercase`. All other eyebrows ("Honest Comparison", "Next Step") use title case in the source. For source consistency, consider making all eyebrow source text title case: "What Ships Free". **Minor cosmetic.**

- [ ] PM-16: **Theme card hover has `transform: translateY(-2px)`.** This is a subtle motion effect on cards in sections 2-5 which should be "static" per spec section 7.4. The hover lift is interactive feedback rather than a scroll animation, so it may be acceptable, but flagging for awareness since the spec says "sections 2-5 render static."

- [ ] PM-17: **layout.tsx missing `alternates.canonical` and `icons`.** Spec S5 mentions adding `alternates: { canonical: '/' }` and `icons` to the layout-level metadata. The `alternates.canonical` is present in `page.tsx:9` (which is correct for the landing page specifically), but not in the layout-level metadata at `layout.tsx:25-41`. The `icons` field is entirely absent. **Low priority -- canonical on the page is more correct anyway; icons can be added later.**

- [ ] PM-18: **JSON-LD missing `description` standalone field.** The `page.tsx:83` JSON-LD uses `description: metadata.description` which resolves correctly. However, the `name` field at line 73 just says `'Yable'` -- the spec mentions `name` should be present (it is), and `description` (it is). This is correct. **Status: acceptable, no fix needed.**
